WebAssembly

WebAssembly: l'ère de naissance d'un nouveau langage de programmation

Qu'est-ce que le WebAssembly?

WebAssembly (abrégé en Wasm) est un format d’instructions binaires pour une machine virtuelle à base de pile. (Définition extrait du site webassembly.org)

Wasm est conçu comme une cible portable de la compilation de langages [de programmation] de haut niveau comme le C, C++, Rust, facilitant le déploiment sur le Web d’application côté client comme serveur.

Le but de la création Wasm

WebAssembly est conçu pour maintenir la nature sans version, testée et rétrocompatible du Web . Les modules WebAssembly pourront appeler et sortir du contexte JavaScript et accéder aux fonctionnalités du navigateur via les mêmes API Web accessibles à partir de JavaScript. WebAssembly prend également en charge les intégrations non Web .

Historique de la création du WebAssembly

-Juin 2015 : annonce de WASM ;
-Création du WebAssembly Community Group avec des représentants des équipes de développement des navigateurs web telles que: Chrome, Edge, Firefox, WebKit.
-Mars 2017 : MVP de l’API WASM (fin des pré-versions expérimentales des navigateurs);
-Novembre 2017 : support de tous les navigateurs majeurs;
-Mars 2019 : annonce de WASI (WebAssembly en dehors du navigateur);
-Décembre 2019 : publication de WASM Core 1.0, une technologie récente qui est disponible dans 4 principaux moteurs de navigation Chrome, Mozilla, safari et Explore.

Objectifs et avantages de la création de ce nouveau bijou dans le monde du web

Rapidité:

    La rapidité est l’un des points clés de WebAssembly. Car le code est rapide à télécharger et à l’exécuter.

Portabilité:

    Cette technologie orientée pour le Web fonctionne quel que soit le Sytème d’Exploitation cible et quelque plateforme matérielle.

Lisibilité

    La lisibilité du code doit favoriser la mise au point et les corrections. Cet objectif reste à réaliser. Dans le fonctionnement actuel, la lisibilité du code repose sur le langage source.

Sécurité:

    Dans l’environnement du Web inter-connecté, les questions de sécurité sont tout de suite mises en avant. Toutefois, l’exécution ne doit pas diminuer les capacités de fonctionnement du navigateur ou mettre même en danger le fonctionnement du Système d’Exploitation hôte. Dans ce cas, l’utilisation systématique d’une mécanique de « bac à sable » s’est donc imposée.

Un plus dans le système: WASI

    Une des limites de WebAssembly est l’impossibilité d’accéder aux ressources locales du client (notamment les fichiers), du fait de l’exécution dans le navigateur, pour raisons de sécurité.
    WASI (WebAssembly System Interface) qui est en développement à l’heure actuelle permettra de s’affranchir de cette limite, en autorisant l’exécution en dehors du navigateur.

Quelques domaines d’utilisation de WebAssembly

Dans le navigateur :

    Vision par ordinateur
    Manipulation d’image / de vidéo
    Jeux vidéos
    Réalité virtuelle / augmentée
    Emulation matérielle
    Cryptage
De façon générale, toute application consommatrice de ressource processeur, de calculs intensifs est concernée.

En dehors du navigateur

    Jeux vidéos
    Exécution de code non vérifié côté serveur
    Application côté serveur
    Applications mobiles hybrides (application mobile liée à un ou plusieurs serveurs)

Les formats sources

WebAssembly n’a pas été prévu pour être programmé directement, mais plutôt comme du code intermédiaire proche des instructions de base des processeurs. Le programmeur dispose donc du code source dans le langage de haut niveau (C, C++ ou Rust), que le compilateur transforme en un code binaire ou texte Wasm très proche de l’assembleur, et c’est ensuite le navigateur qui traduit en code machine pour le processeur dans son bac à sable WebAssembly.
